## Sub-command dispatch (CRITICAL)
Commands are registered in plugin.yml as single words. Sub-commands are passed as args[0].

CORRECT pattern for `/guard spawn|target|move|home`:
```java
@Override
public boolean onCommand(CommandSender sender, Command command, String label, String[] args) {
    if (args.length == 0) { sender.sendMessage("Usage: /guard <spawn|target|move|home>"); return true; }
    switch (args[0].toLowerCase()) {
        case "spawn": return handleSpawn(sender, args);
        case "target": return handleTarget(sender, args);
        case "move":   return handleMove(sender, args);
        case "home":   return handleHome(sender, args);
        default: sender.sendMessage("Unknown sub-command: " + args[0]); return true;
    }
}
```
In onEnable(): `getCommand("guard").setExecutor(this);` — one call, not one per sub-command.
NEVER do `getCommand("guard spawn")` — it will always return null.

## Persistent entity reload
When a plugin saves entity UUIDs to config and reloads on server restart, ALWAYS search the
world for the existing entity by UUID before spawning a new one. Spawning unconditionally creates
duplicate entities every restart:

```java
UUID saved = UUID.fromString(config.getString("entityUUID"));
Entity existing = world.getEntities().stream()
    .filter(e -> e.getUniqueId().equals(saved)).findFirst().orElse(null);
if (existing == null) {
    existing = world.spawn(location, Zombie.class); // only spawn if not found
}
```

## Event listeners
Register every Listener in onEnable() — forgetting this means no events fire:
```java
getServer().getPluginManager().registerEvents(myListener, this);
```
Annotate each handler with `@EventHandler`. Only override `priority` when you specifically need
to run before or after other plugins. Never store a `Player` reference in a field — store the
UUID and resolve it with `Bukkit.getPlayer(uuid)` on use, since players can log out.

## Inventory GUIs
Use a title prefix so your `InventoryClickEvent` handler can recognise your own inventories:
```java
private static final String PREFIX = "MyPlugin: ";

// open
Inventory inv = Bukkit.createInventory(null, 27, PREFIX + "Main Menu");
player.openInventory(inv);

// handle clicks
@EventHandler
public void onInventoryClick(InventoryClickEvent e) {
    if (!e.getView().getTitle().startsWith(PREFIX)) return;
    e.setCancelled(true); // always — prevents item movement and duplication
    // route by full title string, act on e.getSlot() / e.getCurrentItem()
}
```
Inventory size must be a multiple of 9, max 54. Register the listener in onEnable().

## Scheduled tasks
For repeating work use BukkitRunnable, not Java threads:
```java
new BukkitRunnable() {
    @Override public void run() { /* runs on main thread, safe to use Bukkit API */ }
}.runTaskTimer(plugin, 0L, 20L); // delay 0 ticks, repeat every 20 ticks (1 second)
```
Use `runTaskTimerAsynchronously` only for I/O (file/network) — never call Bukkit API from an async task.
Cancel with the returned `BukkitTask` reference or by calling `cancel()` inside the runnable.

## Config management
Call `saveDefaultConfig()` before any `getConfig()` call in `onEnable()`. Always supply a
default so the plugin never throws on a missing key:
```java
int radius = getConfig().getInt("radius", 10);
```
After writing a value, call `saveConfig()` immediately. For a reload command, call
`reloadConfig()` then re-read every cached config value into your fields.
